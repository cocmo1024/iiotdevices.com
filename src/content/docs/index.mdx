## What this site optimizes

<div class="signal-strip">
	<div>
		<strong>Research model</strong>
		<p>Application first, device class second, vendor and protocol after that. This prevents catalog-first selection mistakes.</p>
	</div>
	<div>
		<strong>High-value traffic</strong>
		<p>Device selection, protocol tradeoffs, vendor shortlists, and architecture comparisons tend to attract real project intent.</p>
	</div>
	<div>
		<strong>Long-term edge</strong>
		<p>Reference pages stay useful because device classes and buying questions change more slowly than news cycles or product launches.</p>
	</div>
</div>

## How to use this reference

<Steps>
	1. Begin with the application or site profile that matches the deployment environment.
	2. Narrow the likely device classes before looking at vendors or model names.
	3. Use protocol pages to pressure-test interoperability and data path assumptions.
	4. Finish with comparison workflows that account for support, lifecycle, and rollout realities.
</Steps>
